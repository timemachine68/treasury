# แผนและการออกแบบระบบ: ฝึกซ้อมสอบสัมภาษณ์เข้ามหาวิทยาลัย (นักเรียน ม.6)

> เอกสารฉบับนี้เป็น "แผนงาน + System Design" สำหรับพัฒนาระบบฝึกซ้อมสอบสัมภาษณ์
> ต่อยอดจากระบบ **คลังแนะแนว | แนะให้แนว** (`index.html`) ที่ใช้งานอยู่แล้ว

---

## 1. เป้าหมายและปัญหาที่ต้องการแก้

นักเรียน ม.6 ต้องสอบสัมภาษณ์ใน TCAS หลายรอบ (Portfolio, Quota, กสพท/MMI, รับตรง)
แต่ปัจจุบัน:

- นักเรียนไม่รู้ว่าคณะ/สาขาที่ตนสมัครมักถามอะไร และไม่มีที่ฝึกตอบอย่างเป็นระบบ
- ครูแนะแนวมีเวลาซ้อมสัมภาษณ์จำลอง (mock interview) ให้รายคนได้จำกัด
- ไม่มีบันทึกพัฒนาการ — นักเรียนซ้อมแล้วไม่รู้ว่าดีขึ้นหรือไม่ ครูไม่เห็นภาพรวมทั้งห้อง

**เป้าหมายของระบบ**

1. นักเรียนฝึกตอบคำถามสัมภาษณ์ได้ด้วยตนเอง ทุกที่ ทุกเวลา พร้อมจับเวลาและอัดเสียง/วิดีโอ
2. ครูให้ feedback ตาม rubric มาตรฐานเดียวกัน และเห็นพัฒนาการของนักเรียนเป็นรายคน/รายห้อง
3. จัดคิวซ้อมสัมภาษณ์จำลองกับครูได้สะดวก ลดงานธุรการของครูแนะแนว

## 2. ผู้ใช้งาน (Personas & Roles)

ใช้บัญชีและ role เดิมจากตาราง `profiles` ของคลังแนะแนว — **ไม่ต้องสมัครใหม่**

| Role | สิ่งที่ทำได้ |
|---|---|
| **นักเรียน** | เลือกสาขาเป้าหมาย, ฝึกตอบคำถาม (พิมพ์/อัดเสียง/อัดวิดีโอ), ประเมินตนเอง, จองคิว mock interview, ดู feedback และกราฟพัฒนาการของตนเอง |
| **ครู** | จัดการคลังคำถาม, เปิดตาราง mock interview, ให้คะแนน+feedback ตาม rubric, ดูแดชบอร์ดรายห้อง/รายคน |
| **admin** | ทุกอย่างของครู + จัดการชุดคำถามกลาง, ตั้งค่า rubric, จัดการสิทธิ์ |

## 3. ขอบเขตฟีเจอร์ (แบ่งตามเฟสพัฒนา)

### เฟส 1 — MVP (ใช้งานได้จริงเร็วที่สุด)

1. **คลังคำถามสัมภาษณ์ (Question Bank)**
   - แยกหมวด: คำถามทั่วไป / รายกลุ่มสาขา (แพทย์-พยาบาล, วิศวะ-วิทย์, ครุศาสตร์, บริหาร-มนุษย์ฯ, ศิลปกรรม ฯลฯ) / ภาษาอังกฤษ / MMI
   - แต่ละข้อมี: ตัวคำถาม, แนวทางการตอบ (ซ่อนไว้จนกว่าจะตอบเสร็จ), ระดับความยาก, เวลาแนะนำ
   - ครู/admin เพิ่ม-แก้ไขได้, มาพร้อมชุดคำถามตั้งต้น (seed) ~80–120 ข้อ
2. **โหมดฝึกซ้อมด้วยตนเอง (Solo Practice)**
   - เลือกหมวด → ระบบสุ่มคำถามเป็นเซ็ต (เช่น 5 ข้อ) → จับเวลาต่อข้อ
   - ตอบแบบ **พิมพ์** หรือ **อัดเสียง** (MediaRecorder API) → เก็บไฟล์ใน Supabase Storage
   - จบเซ็ตแล้ว: เปิดแนวทางการตอบให้เทียบ + **ประเมินตนเอง** ตาม rubric ย่อ
3. **Feedback จากครู**
   - นักเรียนกด "ส่งให้ครูตรวจ" ได้ (เลือกส่งเป็นรายเซ็ต)
   - ครูเปิดฟัง/อ่าน แล้วให้คะแนน rubric 4 ด้าน + คอมเมนต์ข้อความ
4. **แดชบอร์ดครู (พื้นฐาน)**
   - รายห้อง: ใครซ้อมแล้วกี่ครั้ง, ใครยังไม่เริ่ม, งานที่รอตรวจ

### เฟส 2 — Mock Interview & พัฒนาการ

5. **จองคิวซ้อมสัมภาษณ์จำลองกับครู** — ครูเปิด slot (วัน/เวลา/จำนวนรับ), นักเรียนจอง, ครูบันทึกผลลง rubric เดียวกันหลังซ้อมจริง
6. **กราฟพัฒนาการ** — คะแนน rubric เฉลี่ยต่อครั้ง เทียบเป็น timeline รายคน
7. **อัดวิดีโอ** (ต่อยอดจากอัดเสียง) + นักเรียนดูย้อนหลังเพื่อสังเกตบุคลิกภาพตนเอง
8. **Checklist เตรียมตัว** — เอกสาร, การแต่งกาย, มารยาท, คำถามที่ควรถามกรรมการกลับ

### เฟส 3 — ส่วนเสริม (ทำเมื่อเฟส 1–2 นิ่งแล้ว)

9. **AI ช่วยให้ feedback เบื้องต้น** จากคำตอบแบบพิมพ์ (ผ่าน Supabase Edge Function เรียก Claude API — ไม่ฝัง API key ในหน้าเว็บ)
10. **โหมดจับคู่เพื่อนซ้อม (Peer Practice)** — เพื่อนเป็นกรรมการ ให้คะแนนกันเอง
11. **รายงานสรุปสำหรับผู้บริหาร/PLC** — export CSV

> ❌ **นอกขอบเขต:** วิดีโอคอลในระบบ (ใช้ซ้อมต่อหน้า/ผ่านช่องทางที่โรงเรียนมีอยู่แล้ว), การวิเคราะห์เสียง-สีหน้าอัตโนมัติ, ระบบสมัครสอบจริง

## 4. Rubric การประเมิน (มาตรฐานเดียวทั้งระบบ)

ให้คะแนน 1–4 ต่อด้าน (สอดคล้องแนวประเมินสัมภาษณ์ของมหาวิทยาลัยส่วนใหญ่):

| ด้าน | สิ่งที่ดู |
|---|---|
| **เนื้อหาคำตอบ** | ตรงคำถาม มีเหตุผล มีตัวอย่างจริง เชื่อมโยงกับสาขาที่สมัคร |
| **การสื่อสาร** | ชัดเจน เรียงลำดับดี น้ำเสียง จังหวะ ไม่วกวน |
| **บุคลิกภาพและความมั่นใจ** | ท่าที การสบตา ความสุภาพ การควบคุมความตื่นเต้น |
| **ทัศนคติและความพร้อม** | รู้จักตนเอง รู้จักสาขา/สถาบัน แรงจูงใจชัดเจน |

- นักเรียนประเมินตนเองด้วย rubric เดียวกัน → ระบบแสดง "ช่องว่าง" ระหว่างมุมมองตนเองกับครู
- เก็บนิยามเกณฑ์ไว้ในตาราง `iv_rubric_criteria` เพื่อให้ admin ปรับถ้อยคำได้โดยไม่ต้องแก้โค้ด

## 5. สถาปัตยกรรมระบบ

### 5.1 แนวทางที่เลือก: แอปแยกไฟล์ ใช้ Supabase โปรเจกต์เดิมร่วมกัน

```
┌──────────────────────┐        ┌──────────────────────────┐
│  index.html          │ ลิงก์ผ่าน │  interview.html (ใหม่)     │
│  (คลังแนะแนว)         │ ─────▶ │  ระบบฝึกซ้อมสัมภาษณ์        │
│  เมนู "ระบบของเรา"     │        │  single-file, Thai UI     │
└──────────┬───────────┘        └────────────┬─────────────┘
           │                                 │
           ▼                                 ▼
┌─────────────────────────────────────────────────────────┐
│                Supabase (โปรเจกต์เดิม)                     │
│  Auth (บัญชีเดิม) · profiles (ตารางเดิม, ใช้ร่วม)            │
│  ตารางใหม่ prefix iv_* · Storage bucket: iv-recordings    │
│  RLS ทุกตาราง · Edge Function (เฟส 3: AI feedback)        │
└─────────────────────────────────────────────────────────┘
```

**เหตุผล**

- `index.html` มีขนาด ~330KB แล้ว การยัดโมดูลใหญ่เพิ่มจะดูแลยากขึ้นเรื่อย ๆ → แยกเป็น `interview.html` แต่คงสไตล์/ฟอนต์/โทนสีเดียวกัน (Sarabun + Mitr, โทนเขียว pine) ให้รู้สึกเป็นระบบครอบครัวเดียวกัน
- ใช้ **Supabase Auth เดิม** → นักเรียน login ครั้งเดียวใช้ได้ทั้งสองระบบ (โดเมนเดียวกัน session ใช้ร่วมกันอัตโนมัติ)
- ลงทะเบียนลิงก์ในตาราง `external_systems` ของคลังแนะแนว → ปรากฏในเมนู "ระบบของเรา" ทันทีโดยไม่ต้องแก้ `index.html`
- ตารางใหม่ใช้ prefix `iv_` กันชนกับตารางเดิมทั้งหมด

### 5.2 เทคโนโลยี

| ส่วน | เทคโนโลยี | หมายเหตุ |
|---|---|---|
| Frontend | HTML/CSS/JS ไฟล์เดียว (เหมือน index.html) | ไม่มี build step, deploy ง่าย (GitHub Pages ได้) |
| Auth + DB | Supabase (โปรเจกต์เดิม) + RLS | ใช้ anon key ฝั่ง client เหมือนเดิม |
| ไฟล์เสียง/วิดีโอ | MediaRecorder API → Supabase Storage | อัดเป็น webm/opus (เสียง ~1MB/นาที) |
| AI (เฟส 3) | Supabase Edge Function → Claude API | เก็บ API key ฝั่ง server เท่านั้น |

## 6. Data Model (ตารางใหม่ทั้งหมดใช้ prefix `iv_`)

```sql
-- หมวด/ชุดคำถาม (เช่น "แพทย์-MMI", "วิศวกรรม", "ภาษาอังกฤษ")
iv_categories      (id, name, name_en, emoji, color, sort_order, is_active)

-- คลังคำถาม
iv_questions       (id, category_id → iv_categories, question_text,
                    guide_answer,          -- แนวทางการตอบ (เปิดหลังตอบเสร็จ)
                    difficulty,            -- easy | medium | hard
                    suggested_seconds,     -- เวลาแนะนำต่อข้อ
                    lang,                  -- th | en
                    created_by → profiles, is_active)

-- เกณฑ์ rubric (แก้ถ้อยคำได้โดยไม่แก้โค้ด)
iv_rubric_criteria (id, code, title, description, sort_order, is_active)

-- 1 ครั้งของการฝึก = 1 session (ทั้ง solo และ mock)
iv_sessions        (id, student_id → profiles,
                    kind,                  -- solo | mock
                    category_id → iv_categories,
                    status,                -- draft | submitted | reviewed
                    submitted_at, reviewed_at, reviewer_id → profiles,
                    teacher_comment, created_at)

-- คำตอบรายข้อใน session
iv_answers         (id, session_id → iv_sessions, question_id → iv_questions,
                    answer_text,           -- กรณีพิมพ์
                    media_path,            -- path ใน Storage กรณีอัดเสียง/วิดีโอ
                    media_kind,            -- none | audio | video
                    seconds_used, order_no)

-- คะแนน rubric: แถวละ 1 เกณฑ์ แยกว่าใครประเมิน
iv_scores          (id, session_id → iv_sessions,
                    criteria_id → iv_rubric_criteria,
                    scorer_role,           -- self | teacher
                    scorer_id → profiles, score int (1–4), note)

-- เฟส 2: ตารางนัด mock interview
iv_slots           (id, teacher_id → profiles, slot_date, start_time, end_time,
                    capacity, location, note, is_open)
iv_bookings        (id, slot_id → iv_slots, student_id → profiles,
                    status,                -- booked | done | cancelled | no_show
                    session_id → iv_sessions)  -- ผูกผลประเมินหลังซ้อมจริง
```

**Storage:** bucket `iv-recordings` (private) โครง path `{{student_id}}/{{session_id}}/{{answer_id}}.webm`

### RLS (หัวใจความปลอดภัย)

- `iv_categories`, `iv_questions`, `iv_rubric_criteria`: ทุกคนที่ login อ่านได้ (ยกเว้น `guide_answer` — ดูหมายเหตุ¹) / เขียนได้เฉพาะ teacher+admin
- `iv_sessions`, `iv_answers`: นักเรียน CRUD ได้เฉพาะแถวของตนเอง; teacher/admin อ่านได้ทั้งหมด และ update ได้เฉพาะฟิลด์รีวิว
- `iv_scores`: `scorer_role='self'` เขียนได้เฉพาะเจ้าของ session; `scorer_role='teacher'` เขียนได้เฉพาะ teacher/admin
- `iv_slots`: teacher จัดการของตนเอง, ทุกคนอ่าน slot ที่ `is_open`
- `iv_bookings`: นักเรียนจอง/ยกเลิกของตนเอง, teacher เห็นของ slot ตนเอง
- Storage: policy ให้เข้าถึงเฉพาะไฟล์ใต้โฟลเดอร์ `student_id` ของตนเอง + teacher/admin อ่านได้ทั้ง bucket

> ¹ ฝั่ง client ซ่อน `guide_answer` จนกว่าจะตอบเสร็จก็เพียงพอสำหรับบริบทโรงเรียน (ไม่ใช่ข้อสอบลับ) — ถ้าต้องการเข้มขึ้นค่อยแยกตาราง `iv_question_guides` พร้อม RLS ภายหลัง

## 7. User Flows หลัก

### นักเรียน — ฝึกซ้อมเอง
```
เข้าระบบ (บัญชีคลังแนะแนว) → เลือกหมวด (เช่น พยาบาล)
→ เริ่มเซ็ต 5 ข้อ: เห็นคำถาม + นาฬิกาถอยหลัง → เลือกพิมพ์ / กดอัดเสียง
→ ครบเซ็ต: เปิดแนวทางการตอบ เทียบกับของตัวเอง → ประเมินตนเอง 4 ด้าน
→ (เลือกได้) ส่งให้ครูตรวจ → เห็นสถานะ "รอครูตรวจ"
```

### ครู — ตรวจและติดตาม
```
เข้าระบบ → แดชบอร์ด: การ์ด "รอตรวจ (n)" / ภาพรวมรายห้อง
→ เปิดงานนักเรียน: ฟังเสียง/อ่านคำตอบ ข้อต่อข้อ → ให้คะแนน rubric + คอมเมนต์
→ กดส่งคืน → นักเรียนได้รับ feedback, กราฟพัฒนาการอัปเดต
```

### เฟส 2: นัด mock interview
```
ครูเปิด slot (เช่น พฤ 14:30–16:30 รับ 6 คน ห้องแนะแนว)
→ นักเรียนเห็นตารางว่าง → กดจอง → ซ้อมจริงต่อหน้า
→ ครูกรอก rubric ในระบบทันทีหลังซ้อม → ผลรวมอยู่ใน timeline เดียวกับ solo
```

## 8. โครงหน้าจอ (UI Sitemap ของ `interview.html`)

ใช้แท็บแฟ้มแบบเดียวกับคลังแนะแนว:

| แท็บ | นักเรียนเห็น | ครูเห็น |
|---|---|---|
| 🏠 หน้าแรก | สาขาเป้าหมาย, ปุ่มเริ่มฝึก, สถิติของฉัน | การ์ดรอตรวจ, สรุปรายห้อง |
| 🎯 ฝึกซ้อม | เลือกหมวด → เข้าห้องซ้อม (หน้าจอโฟกัส ไม่มีสิ่งรบกวน) | — |
| 📋 ผลของฉัน / งานตรวจ | ประวัติ + feedback + กราฟพัฒนาการ | คิวงานรอตรวจ + หน้าตรวจ |
| 📅 นัดซ้อม (เฟส 2) | จอง/ยกเลิกคิว | เปิด-ปิด slot, รายชื่อผู้จอง |
| 🗂 คลังคำถาม | — | จัดการหมวด/คำถาม |
| ✅ เตรียมตัว (เฟส 2) | checklist + เกร็ดความรู้ | แก้ไขเนื้อหา checklist |

## 9. แผนดำเนินงาน

| ช่วง | งาน | ผลลัพธ์ |
|---|---|---|
| สัปดาห์ 1 | สร้างตาราง `iv_*` + RLS + bucket, โครง `interview.html` (login ร่วม, แท็บ, สไตล์), seed หมวด+คำถามตั้งต้น | เข้าระบบได้ เห็นคลังคำถาม |
| สัปดาห์ 2 | ห้องซ้อม: สุ่มเซ็ต, จับเวลา, พิมพ์คำตอบ, ประเมินตนเอง, ส่งครู | นักเรียนฝึกครบวงจร (แบบพิมพ์) |
| สัปดาห์ 3 | อัดเสียง + Storage, หน้าตรวจของครู + rubric, แดชบอร์ดพื้นฐาน → **ปล่อย MVP ให้ ม.6 ทดลองใช้ 1 ห้อง** | MVP ใช้จริง |
| สัปดาห์ 4–5 | เก็บ feedback ปรับปรุง, เฟส 2 (นัด mock, กราฟพัฒนาการ, checklist, อัดวิดีโอ) | ใช้เต็มระดับชั้น |
| ถัดไป | เฟส 3 ตามความพร้อม (AI feedback, peer practice, รายงาน CSV) | ส่วนเสริม |

## 10. ความเสี่ยงและแนวทางรับมือ

| ความเสี่ยง | แนวทางรับมือ |
|---|---|
| พื้นที่ Storage เต็ม (ไฟล์เสียง/วิดีโอ) | เริ่มจากเสียงอย่างเดียว (webm/opus ~1MB/นาที), จำกัดความยาวต่อข้อ (เช่น 3 นาที), ตั้งนโยบายลบไฟล์เก่ากว่า 1 ปีการศึกษา |
| ความเป็นส่วนตัวของไฟล์อัดนักเรียน | bucket private + RLS รายคน, แจ้งนักเรียนชัดเจนว่าครูแนะแนวเท่านั้นที่เห็น, ไม่มีการแชร์สาธารณะ |
| มือถือบางรุ่นอัดเสียงไม่ได้ | ตรวจ `MediaRecorder` ก่อน — ถ้าไม่รองรับให้ fallback เป็นพิมพ์คำตอบเสมอ |
| ครูมีภาระตรวจมากช่วงใกล้ TCAS | การประเมินตนเอง + แนวทางการตอบช่วยให้ solo mode มีคุณค่าแม้ครูยังไม่ตรวจ; แดชบอร์ดช่วยจัดลำดับ; เฟส 3 ให้ AI ช่วยกรองเบื้องต้น |
| คำถามในคลังล้าสมัย/ไม่ตรงคณะ | ให้ครูทุกคนเพิ่มคำถามได้ + ปุ่ม "รายงานคำถาม" จากนักเรียน |

## 11. ตัวชี้วัดความสำเร็จ (หลังใช้ 1 ภาคเรียน)

- นักเรียน ม.6 ≥ 70% ฝึกซ้อมอย่างน้อย 3 เซ็ตก่อนช่วงสัมภาษณ์จริง
- คะแนน rubric เฉลี่ยของนักเรียนที่ซ้อม ≥ 3 ครั้ง มีแนวโน้มเพิ่มขึ้น
- ครูใช้เวลาจัดคิว mock interview ลดลง (จากนัดปากเปล่า/กระดาษ → ระบบ)
- แบบสอบถามความมั่นใจก่อน-หลัง (ผูกกับระบบสำรวจของคลังแนะแนวเดิมได้)
